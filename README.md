<summary><b>🧪 Jenis-Jenis Testing</b></summary>

### ⬛ Black Box Testing
Pengujian yang dilakukan **tanpa mengetahui struktur internal kode**. Tester hanya fokus pada input dan output yang diharapkan berdasarkan requirement.

| Teknik | Deskripsi | Penggunaan Saya |
|--------|-----------|-----------------|
| Equivalence Partitioning | Membagi input menjadi kelompok yang valid dan tidak valid | Digunakan saat membuat test case untuk form input |
| Boundary Value Analysis | Menguji nilai di batas minimum dan maksimum | Digunakan untuk testing limit transaksi di Bank BRI |
| Decision Table | Kombinasi kondisi dan aksi yang diharapkan | Digunakan untuk testing fitur insurance claim |
| State Transition | Menguji perubahan status sistem | Digunakan untuk testing status kartu kredit (active/inactive/blocked) |
| Exploratory Testing | Testing tanpa script, berdasarkan intuisi dan pengalaman | Dilakukan untuk menemukan bug di luar skenario yang sudah ada |

---

### ⬜ White Box Testing
Pengujian yang dilakukan **dengan mengetahui struktur internal kode**. Fokus pada logika, kondisi, dan alur kode.

| Teknik | Deskripsi | Penggunaan Saya |
|--------|-----------|-----------------|
| Statement Coverage | Memastikan setiap baris kode dieksekusi | Digunakan saat review automation script |
| Branch Coverage | Memastikan setiap cabang kondisi (if/else) diuji | Diterapkan saat menulis assertion di Playwright |
| Path Coverage | Menguji semua kemungkinan jalur eksekusi | Diterapkan untuk alur transaksi kompleks |
| Unit Testing | Menguji fungsi/unit terkecil secara terisolasi | Berkolaborasi dengan developer untuk validasi unit |

---

### 🔲 Grey Box Testing
Kombinasi black box dan white box — tester **memiliki pengetahuan parsial** tentang internal sistem, seperti struktur database atau API contract.

> Contoh penggunaan saya: Saat melakukan API testing dengan Postman, saya memverifikasi response body sekaligus memvalidasi data yang tersimpan di database menggunakan DBeaver — ini adalah pendekatan grey box karena saya tahu skema database-nya.

---

<summary><b>📐 Testing Strategy & Methodology</b></summary>

### 🔄 Agile Testing
Saya bekerja dalam **Agile/Scrum environment** di semua perusahaan. Testing bukan aktivitas terpisah di akhir sprint — melainkan **terintegrasi sepanjang siklus development**.

**Praktik yang saya terapkan:**
- Terlibat sejak sprint planning untuk memahami acceptance criteria
- Membuat test scenario bersamaan dengan development (shift-left testing)
- Automation dijalankan di setiap PR melalui CI/CD pipeline
- Demo hasil testing di sprint review
- Retrospective untuk evaluasi dan perbaikan proses QA

---

### 🔺 Testing Pyramid
Saya mengikuti prinsip **Testing Pyramid** dalam menentukan proporsi jenis test:

```
        /\
       /  \   E2E Tests (sedikit, mahal, lambat)
      /----\
     /      \ Integration Tests (sedang)
    /--------\
   /          \ Unit Tests (banyak, murah, cepat)
  /____________\
```

> **Penerapan saya:** Di Lion Parcel, saya memastikan automation suite memiliki lebih banyak API/integration test dibanding E2E test, karena lebih cepat, lebih stabil, dan lebih mudah di-maintain.

---

### 🎯 Risk-Based Testing
Strategi yang saya gunakan untuk memprioritaskan testing berdasarkan **risiko bisnis dan teknis**:

1. **Identifikasi risiko** — Fitur mana yang paling kritikal jika gagal?
2. **Analisis dampak** — Seberapa besar dampak ke user dan bisnis?
3. **Tentukan likelihood** — Seberapa besar kemungkinan area ini punya bug?
4. **Prioritaskan** — High risk + high impact = prioritas tertinggi
5. **Alokasikan effort** — Fokus automation pada area berisiko tinggi

---

### 🔄 Shift-Left Testing
Pendekatan di mana testing dimulai **sedini mungkin** dalam siklus development, bukan di akhir.

**Praktik konkret saya:**
- Review requirement dan desain sebelum coding dimulai
- Menulis test case bersamaan dengan developer menulis kode
- Identifikasi ambiguitas requirement sejak awal
- Feedback loop cepat antara QA dan developer

---

### 🚦 Severity vs Priority

Ini adalah dua konsep yang **sering disamakan tapi berbeda** — salah satu pertanyaan favorit interviewer.

> **Severity** = Seberapa parah dampak bug terhadap **fungsi sistem**  
> **Priority** = Seberapa cepat bug tersebut **harus diperbaiki**

---

#### 📊 Level Severity

| Level | Deskripsi | Contoh |
|-------|-----------|--------|
| **Critical** | Sistem crash atau fitur utama tidak bisa digunakan sama sekali | Aplikasi error saat login, tidak bisa melakukan transaksi |
| **Major** | Fitur utama tidak berjalan sesuai ekspektasi, ada workaround | Data transaksi tidak tersimpan, tapi bisa dicoba ulang |
| **Minor** | Fitur berjalan tapi ada perilaku yang tidak sesuai | Filter sorting tidak bekerja dengan benar |
| **Trivial** | Masalah kosmetik atau UI kecil, tidak mempengaruhi fungsi | Typo pada label, warna tombol tidak sesuai desain |

---

#### 📊 Level Priority

| Level | Deskripsi | Contoh |
|-------|-----------|--------|
| **Critical** | Harus diperbaiki sekarang, menghentikan semua aktivitas lain | Bug di halaman checkout yang menghentikan semua transaksi |
| **High** | Harus diperbaiki di sprint/release ini | Fitur utama tidak berjalan untuk sebagian besar user |
| **Medium** | Bisa diperbaiki di sprint berikutnya | Fitur sekunder tidak bekerja optimal |
| **Low** | Bisa dijadwalkan kapan saja | Typo, perubahan warna, perbaikan kosmetik kecil |

---

#### 🔀 Kombinasi Severity & Priority (Yang Paling Penting Dipahami)

| Kombinasi | Penjelasan | Contoh Nyata |
|-----------|-----------|--------------|
| **High Severity + High Priority** | Bug kritis yang harus segera diperbaiki | Fitur login tidak bisa digunakan |
| **High Severity + Low Priority** | Bug parah tapi tidak mendesak | Bug di fitur lama yang sudah tidak digunakan user |
| **Low Severity + High Priority** | Bug kecil tapi harus segera diperbaiki karena alasan bisnis | Typo pada nama CEO di halaman utama sebelum press release |
| **Low Severity + Low Priority** | Bug kecil dan tidak mendesak | Warna icon tidak sesuai style guide di halaman settings |

> 💡 **Contoh dari pengalaman saya:** Di Bank BRI, saya pernah menemukan inkonsistensi tampilan angka saldo di mobile — severity-nya Minor (tidak mempengaruhi fungsi), tapi priority-nya High karena akan ada demo ke direksi keesokan harinya. Ini contoh klasik Low Severity + High Priority.
